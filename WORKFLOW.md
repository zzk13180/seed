# LLM 驱动开发流程 (LLM-Driven Development Workflow)

本流程旨在建立一套标准化的交互协议，利用 LLM (Large Language Model) 高效构建符合 **Golden Seed** 标准的代码库。

## 核心理念
**"以示例为核心 (Example-Centric)"**：利用 LLM 强大的模仿能力（Few-Shot Learning），通过提供高质量的参考代码（Golden Seed）来隐式传递复杂的工程规范，而非依赖冗长的自然语言规则。

## 流程步骤

### 1. 种子培育 (Seed Cultivation)
在规模化开发前，需构建核心场景的“黄金种子”作为标准参照物。
*   **构建对象**：涵盖 UI 组件、业务逻辑、测试用例等核心开发场景。
*   **质量标准**：种子代码必须严格遵循 `z-llm-friendly-architecture.md` 规范，具备生产级质量（包含完整类型、错误处理及测试）。
*   **作用**：确立代码风格与架构模式，作为 LLM 生成新代码的模仿蓝本。

### 2. 上下文构建 (Context Setup)
每次交互必须构建完整的上下文环境，确保 LLM 在受控范围内工作：
*   **规则层 (System Prompt)**：加载全局工程约束与负面清单（参考 `PROMPT.md`）。
*   **范例层 (Golden Seed)**：显式注入与当前任务同类型的“种子代码”，提供具体的实现参照。
*   **约束层 (Negative Constraints)**：确保核心的负面约束（如禁止隐式依赖）在上下文中生效。

### 3. 任务下发 (Task Prompting)
采用结构化框架描述任务，确保指令清晰无歧义：
*   **Role (角色)**：激活 System Prompt 中定义的工程师角色。
*   **Context (语境)**：指定参考的实现模式（即 Golden Seed）。
*   **Task (目标)**：定义明确的功能需求、输入与输出。
*   **Constraint (边界)**：重申关键的架构约束（如文件结构、依赖方式）。

### 4. 生成与验证 (Generation & Verification)
*   **生成**：执行 LLM 代码生成。
*   **多维验证**：
    1.  **静态检查**：Lint、Type Check 通过。
    2.  **架构检查**：是否引入隐式依赖？是否符合同位原则？是否过度抽象？
    3.  **功能检查**：单元测试通过。

### 5. 迭代反馈 (Feedback Loop)
当生成代码不符合规范时，执行**修正输入**而非修正输出：
1.  **禁止手动修正**：不要直接修改生成代码，这会掩盖 Prompt 或 Seed 的缺陷。
2.  **指出偏差**：向 LLM 指出具体违反的规范点，要求重写。
3.  **溯源优化**：
    *   若模式模仿错误 -> **优化种子代码**（使其特征更显著）。
    *   若违反硬性约束 -> **优化 System Prompt**（追加负面约束）。

## 最佳实践
*   **小步迭代**：将复杂任务拆解为接口定义、逻辑实现、测试生成三个独立步骤，避免上下文丢失。
*   **代码即文档**：优先通过重构代码逻辑来提升可读性，而非依赖注释解释复杂逻辑。
